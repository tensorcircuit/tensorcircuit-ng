# Backend and Performance

Use this file for backend-wrapper semantics, tracing and vectorization, XLA execution, hot-path structure, and dtype behavior. Use `contraction_and_benchmarking.md` for tensor-network path search and slicing.

## Backend wrapper semantics

- Add a missing tensor primitive to the backend abstraction and cover it with backend-fixture tests instead of introducing a backend-specific call in core logic.
- `tc.backend.vmap` selects batched arguments with `vectorized_argnums`, not JAX-style `in_axes`; use a tuple such as `vectorized_argnums=(0, 1)` for multiple batched arguments.
- `lexsort(keys)` follows NumPy semantics: the last key is primary. Test every custom backend implementation against that ordering.
- When a primitive can be expressed cleanly from existing backend operations, prefer that composition over a backend-specific escape hatch.

## Tracing, vectorization, and timing

- Keep compiled shapes static. Represent dynamically populated buffers with masks and functional updates rather than tensor-dependent shape changes or in-place mutation.
- JIT the outer computation step, warm it with production shapes, and synchronize asynchronous JAX results with `.block_until_ready()` before timing.
- Use `scan` or chunked streaming when `vmap` would materialize all intermediates. For layered VQE, `scan` is often a fast-compile mode while unrolling gives better steady-state throughput; report them as separate modes and compare total-call break-even with `warmup + (N - 1) * run`.
- For an exact finite-depth 1D bra–MPO–ket contraction, consider scanning fixed-shape spatial blocks with the complete cut tensor as the carry. If layer `l` contributes operator-Schmidt ranks `χ_l^ket` and `χ_l^bra` across the cut and the MPO bond is `D`, the carry has `D * prod_l(χ_l^ket * χ_l^bra)` elements; `D * χ**(2L)` applies only when one uniform rank-`χ` gate crosses the cut in every layer. Thus an RZZ circuit with `χ=2` can be practical while a generic two-qubit gate with rank up to 4 can be exponentially harder. Search separate fixed paths for the caps and repeated bulk, keep boundary-axis order stable, and validate values plus elementwise gradients against a full contraction on smaller systems; no truncation is implied.
- Wrap the spatial block body in `jax.checkpoint` when differentiating a long scan. Without rematerialization, the scan transpose can retain one block's contraction residuals per spatial step; block-level rematerialization can sharply reduce peak memory and may also improve runtime by reducing memory traffic. Choose block size empirically because smaller blocks simplify compilation but increase the number of full carry transfers.
- Use local causal windows only after deriving the cone from the ordered gate schedule. Brick-wall layers and commuting nearest-neighbor sweeps can give a window controlled by circuit depth, but one sequential ladder of generic noncommuting gates can propagate support across `O(n)` sites. When a bounded window is actually valid, scan over local terms and optionally partition independent windows across one-controller-per-GPU processes.
- For many perfect-sampling shots on a fixed circuit, benchmark a jitted, vmapped `perfect_sampling` over status seeds or fixed-size chunks. This avoids the per-shot Python dispatch used by `Circuit.sample(batch=..., allow_state=False)`; tune the contraction-search budget separately because search can dominate staging.
- Prebuild static contraction paths, sparse Hamiltonians, and other structural objects outside timed value-and-gradient callables. Separate search, compile/warmup, first execution, and steady-state timing.
- Simulator comparisons must match circuit, dtype, device, and measurement algorithm and must verify values and gradients first. State explicitly when one path uses sparse-Hamiltonian expectation rather than a Pauli-term loop.
- `expectation_ps` defaults to `reuse=True`; repeated terms on one circuit do not each reconstruct the state unless reuse is explicitly disabled.
- When `DistributedContractor` loads a precomputed path, contraction only requires tensors in the same stable node order. Stable-sort the nodes directly instead of regenerating `input_sets`, `output_set`, and `size_dict` inside every JAX trace. Keep more invasive direct-tensor or contraction-instrumentation hooks in experiment harnesses unless they show a material reusable benefit; modest lowering reductions do not address a dominant XLA backend compile.
- For very large fixed-path gradients, report Python/JAX lowering separately from XLA compilation. Removing tensor-network object work can shorten lowering while leaving a much larger XLA code-generation bottleneck unchanged.
- A JAX persistent compilation-cache hit is not equivalent to an in-process warm call. A new process still constructs and traces the Python function, lowers enough IR to compute the cache key, reads and decompresses the entry, and calls `backend.deserialize_executable(...)` to create a process-local loaded executable; device-context/module initialization, allocations, transfers, and the first synchronized execution can add more time. Time lowering, cache retrieval/deserialization, first execution, and steady execution separately. Use `JAX_LOG_COMPILES=1` and `JAX_EXPLAIN_CACHE_MISSES=1` to distinguish a real hit from recompilation; the mere presence of a cache file proves neither key compatibility nor a hit.

## XLA and GPU execution

- Set `XLA_FLAGS` before Python initializes JAX and report every nondefault flag. Optimization levels are workload-dependent; lowering `--xla_backend_optimization_level` is not a monotonic compile-time improvement.
- For GPU contractions with expensive first execution, A/B test `--xla_gpu_autotune_level=0`. TensorCircuit's `gpu_memory_share(True)` already sets `XLA_PYTHON_CLIENT_PREALLOCATE=false`, so disabling autotune is usually the independent variable; unsetting only the shell variable does not restore preallocation after TensorCircuit configures JAX.
- `--xla_gpu_enable_llvm_module_compilation_parallelism=true` partitions final LLVM device-module compilation; the partition count is not a GPU count and does not invoke inter-GPU communication. It can reduce late backend compilation when the CUDA/XLA installation supports the required provider and linker, but it does not parallelize earlier HLO, layout, fusion, or indexing-map passes. Treat `--xla_disable_hlo_passes=fusion` as a diagnostic flag: tested contraction workloads compiled or ran worse without a memory benefit.
- Installing `libnvJitLink` at runtime does not enable an XLA plugin that was built with libnvJitLink/libNvPtxCompiler support disabled. Official OSS builds can contain support stubs that still reject the runtime flag; enabling the library route requires rebuilding the CUDA PJRT/plugin with the corresponding XLA Bazel build settings.
- A visible `ptxas` plus the CUDA linker executable `nvlink` can provide a subprocess route for LLVM-module parallel compilation without the in-process libraries. This `nvlink` is unrelated to hardware NVLink and requires no GPU-to-GPU NVLink fabric. Verify provider discovery and CPU utilization, then A/B test cold compile time: large contraction graphs can remain dominated by earlier serial optimization and show no material speedup even when the flag is accepted.
- XLA logical buffer analysis and contraction width underpredict actual GPU peak memory because execution also needs transposes, workspaces, caches, and contiguous allocator blocks. Measure peak device memory in a fresh process for serious capacity decisions; `jax.clear_caches()` need not release autotune or device-library state.
- Diagnose single-process `pmap` and one-controller-per-GPU execution separately because they exercise different NCCL communicator, CUDA-context, and bootstrap paths. Test both pure `pmap(identity)` with a single-GPU-resident input and `pmap(lax.psum)` with host and explicitly sharded inputs, then inspect every addressable shard; a collective can finish while wrong first-stage resharding makes all results numerically false. `NCCL_P2P_DISABLE=1` only changes NCCL transport and does not disable PJRT/CUDA peer copies. If official `all_reduce_perf` stalls on `P2P/direct pointer` but passes via `SHM/direct/direct`, leave SHM enabled and independently test raw `cudaMemcpyPeer` with peer access enabled versus disabled: a driver/platform can report peer access as available while direct copies hang or silently return zeros, whereas the non-peer fallback is correct. On bare-metal Linux, also inspect each GPU IOMMU group's domain type; a translated domain such as `DMA-FQ` is a high-priority A/B against an IOMMU-disabled boot because CUDA PCIe P2P may require identity/no-IOMMU routing. Host-originated or explicit per-device initial placement can avoid the bad copy; disabling CUDA peer access after PJRT initialization can be a diagnostic/process-local workaround, not a framework default. Match the complete JAX/JAXLIB/PJRT plugin as well as NCCL when comparing hosts because a communicator regression can persist after only NCCL is downgraded. Host reduction neither fixes bad input sharding nor provides general device-side SPMD semantics; reserve it for small independent results. Shutdown watcher warnings after a verified final barrier remain teardown noise.

## Hamiltonian MVP and ODE hot paths

- In JAX/Diffrax vector fields for Pauli sums, prefer a matrix-free callable such as `H_mvp(y)` over sparse COO matmul. Nested solve tracing and reverse-mode AD can otherwise carry large COO index constants through `jit`, `grad`, and shape evaluation.
- Keep MVP closures pure under JAX transformations. Do not cache tensors created during tracing in mutable Python state, because tracers can leak across Diffrax's nested transforms.
- Validate the matrix-free and sparse paths numerically before claiming the ODE-specific compile or runtime gain.

## Dtype and long-lived constants

- If a registered object can outlive one trace, normalize raw NumPy inputs to the active TensorCircuit dtype before they become closed-over constants; a late cast in `__call__` may occur after JAX has captured the array.
- Disable TF32 or equivalent reduced-precision GPU modes before trusting complex64 cross-backend comparisons. `jax_default_matmul_precision="highest"` alone may still leave small complex TensorCircuit contractions on a reduced-precision NVIDIA path; verify state norms or a CPU reference and use the vendor TF32 override when needed.
- JAX disables `int64` by default under TensorCircuit complex64 mode. Bit-packed sparse algorithms that require `int64` should use the high-precision test fixture or fail clearly when x64 is unavailable; coefficient precision is a separate concern.
